Kubernetes Networking Deep Dive

February 2025 Kubernetes Networking

Introduction

Kubernetes networking can be complex. This guide breaks down the key concepts: Pod networking, Services, Ingress, and Network Policies. Understanding these fundamentals is crucial for running production workloads.

Pod Networking Basics

Every Pod gets its own IP address. Containers within a Pod share the network namespace and can communicate via localhost.

kubectl get pods -o wide
NAME        READY   IP            NODE
nginx-pod   1/1     10.244.1.5    worker-1
redis-pod   1/1     10.244.2.3    worker-2

Key insight: Pods can communicate with each other using their IP addresses directly, without NAT. This is a fundamental requirement of the Kubernetes networking model.

Service Types

Services provide stable networking endpoints for Pods:

  • ClusterIP - Internal cluster communication (default)
  • NodePort - Exposes service on each node's IP at a static port
  • LoadBalancer - Provisions an external load balancer (cloud providers)
  • ExternalName - Maps to an external DNS name
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  type: ClusterIP
  selector:
    app: my-app
  ports:
  - port: 80
    targetPort: 8080

Network Policies

Network Policies allow you to control traffic flow at the IP/port level. By default, pods are non-isolated and accept traffic from any source.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

Best practice: Start with a deny-all policy and explicitly allow required traffic. This follows the principle of least privilege.

CNI Plugins

Container Network Interface (CNI) plugins implement the networking model:

  • Calico - Popular choice with Network Policy support
  • Cilium - eBPF-based, advanced observability
  • Flannel - Simple overlay network
  • Weave Net - Easy setup, mesh networking

Conclusion

Understanding these networking primitives is essential for running production Kubernetes clusters securely. Start with the basics, then gradually implement Network Policies to lock down your cluster.
